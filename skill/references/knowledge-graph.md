# KnowledgeGraph Contract

The project has two graph layers. Keep them conceptually separate.

## 1. Content Navigation Graph

Markdown frontmatter is ingested into the ordinary `links` table:

```yaml
prerequisites: [graph-representation, priority-queue]
related: [bellman-ford, bfs]
```

- `prerequisites` points from the current node to a node it depends on.
- `related` is lateral navigation only; it does not establish a parent/child
  relationship in KnowledgeGraph.
- Values are exact existing slugs, not titles, paths, URLs, or generated ids.
- Do not add self-links, duplicate values, or a prerequisite cycle.
- Backlinks are generated by ingest. Do not maintain a second manual backlink
  list in Markdown.

This graph supports search, backlinks, and reading navigation. It is not a
replacement for the explicit KG edge store.

## 2. Explicit KnowledgeGraph DAG

The merged KG module uses these concepts:

| Concept | Meaning |
| --- | --- |
| `kg_question` | A problem/root context with stable `questionId`, root node, category, and scope. |
| `kg_edge` | A directed dependency edge. `parent_node_id -> child_node_id` means the parent depends on the child. |
| `kg_mastery` | Current learner state for a node: `UNKNOWN`, `LEARNING`, `FRAGILE`, or `MASTERED`. |
| `kg_mastery_event` | Append-only PASS/FAIL evidence tied to a quiz item. |
| `kg_proposal` | Pending AI or heuristic tree proposal. It is not active until confirmed. |

Every KG endpoint must resolve to a real Markdown-backed node. A database-only
node is invalid because ingest may rebuild the ordinary node table. A shared
prerequisite (DAG diamond) is valid; copied duplicate nodes are not required.

### Edge and scope invariants

- Active edges have existing parent and child nodes.
- `GLOBAL` edges participate in every question tree.
- `PROBLEM_LOCAL` edges participate only in their `scopeQuestionId` tree.
- A new edge is rejected if `parent == child` or if an existing path already
  reaches `parent` from `child`. The returned error is `CycleDetected` with a
  readable path.
- Cycle checks include the active edges visible to the scope. Do not bypass
  them with direct SQL or by writing a frontmatter guess.
- A question root must exist before prerequisites are appended. A materialized
  tree must be reachable from that root; an unattached node is not a valid
  prerequisite.

## 3. Safe Agent Workflow

1. Resolve the active content root and build a slug/title index.
2. Read the target node and its existing prerequisites, related links, and KG
   snapshot before proposing changes.
3. Read every affected Markdown node in full, including examples and suggested
   links. Classify unfamiliar terms as a local explanation, an existing-node
   link, or a new reusable node before editing the graph.
4. Create missing prerequisite Markdown nodes first. Index them before linking.
5. For a tree request, call `POST /api/kg/proposals/prerequisite-chain`.
   Inspect the returned nested `PrerequisiteSpec`, reuse existing node ids, and
   edit or reject guesses before confirmation.
6. Confirm with `POST /api/kg/proposals/{proposalId}/confirm` only after the
   proposal passes slug, endpoint, direction, scope, duplicate, and cycle
   checks. Use a unique `commandId` for idempotent retries.
7. For a direct append, use `POST /api/kg/questions/{qid}/prerequisites` with
   `GLOBAL` or `PROBLEM_LOCAL` explicitly selected.
8. Read `GET /api/kg/questions/{qid}/tree` or
   `GET /api/kg/nodes/{nodeId}/subtree` and verify root, layers, edges,
   `sharedNodeIds`, and mastery values.
8. If a write fails, do not “repair” it by adding a guessed node. Resolve the
   exact missing endpoint or correct the proposal.

For Markdown-only changes, run:

```powershell
python skill/scripts/validate_knowledge_graph.py <active-content-root>
```

This catches duplicate slugs, missing targets, self-links, prerequisite
cycles, and accidental runtime mastery fields before ingest.

AI may propose titles and relationships, but proposal confirmation and graph
invariants are deterministic application responsibilities.

## 4. Mastery-Driven Learning

Mastery is runtime state. Never put it in Markdown frontmatter.

The offline rule engine applies:

- PASS: `score = min(1, score + 0.20)`, reset `fail_streak`.
- FAIL: `score = max(0, score - 0.25)`, increment `fail_streak`.
- `MASTERED`: score >= `0.80` and the most recent three verdicts contain no
  FAIL. A node can require four PASS events from zero to reach the threshold.
- A weak candidate is `FRAGILE` or `LEARNING` with score < `0.60`.
- Initial state is `UNKNOWN`, score `0.0`, attempts `0`.

After `POST /api/kg/verifications`, use `suggestedNext`: it returns up to three
direct prerequisites with the lowest score. For a quiz gap, use
`GET /api/kg/quizzes/{quizId}/gap`; its priority is
`(1 - score) * (1 + blocksCount)`. Bottlenecks aggregate weak prerequisites
across active question trees and should be used to decide what to reinforce
next, not to rewrite node taxonomy automatically.

## 5. Relevant Endpoints

| Operation | Endpoint | Agent use |
| --- | --- | --- |
| List roots | `GET /api/kg/questions` | Find active problem contexts |
| Read tree | `GET /api/kg/questions/{qid}/tree` | Verify scoped reachability |
| Append tree | `POST /api/kg/questions/{qid}/prerequisites` | Deterministic, cycle-checked write |
| Propose tree | `POST /api/kg/proposals/prerequisite-chain` | AI/heuristic proposal only |
| Confirm proposal | `POST /api/kg/proposals/{pid}/confirm` | Atomic materialization after review |
| Record evidence | `POST /api/kg/verifications` | Update mastery from PASS/FAIL |
| Diagnose gap | `GET /api/kg/quizzes/{qid}/gap` | Choose weakest prerequisite |
| List bottlenecks | `GET /api/kg/bottlenecks` | Prioritize cross-question remediation |
| Export graph | `GET /api/kg/export3d?root=...` | Read-only 3D view with layers/mastery |
| Detach edge | `DELETE /api/kg/edges/{eid}` | Soft-reject an incorrect active edge |

## 6. Preflight Checklist

Before reporting a node or KG change as complete:

- [ ] Active content root is correct.
- [ ] Every link value is an existing exact slug.
- [ ] `prerequisites` direction is current -> dependency.
- [ ] `related` is not being used as a parent edge.
- [ ] No self-link, duplicate, orphan endpoint, or cycle exists.
- [ ] AI proposals were confirmed only after inspection.
- [ ] Scope is explicitly `GLOBAL` or `PROBLEM_LOCAL` with a question id.
- [ ] Tree/subtree snapshot was read after the write.
- [ ] Mastery was updated only through verification evidence.
- [ ] The relevant backend/Kotlin contract tests pass.
